## Context

The local ingestion pipeline creates a `<hash>.chaptermarks.txt` sidecar file for each processed M4A episode, representing chapter markers generated by Ollama in the standard FFmpeg metadata format (`FFMETADATA1`). It also embeds these chapters in the audio container. However, network-streaming podcast clients often fail to read embedded container chapters or require fetching the entire file first. Exposing the chapters via an external JSON file and linking it via the Podcasting 2.0 `<podcast:chapters>` element in the RSS feed allows clients to fetch chapters independently and display them instantly.

## Goals / Non-Goals

**Goals:**
*   Implement a new endpoint `GET /chapters/{filename}` in the cloud FastAPI application (`cloud/app.py`).
*   On-demand parse `<basename>.chaptermarks.txt` files and return them in the official Podcast Chapters JSON format (MIME: `application/json+chapters` or `application/json`).
*   Include the `<podcast:chapters>` XML element under each feed item in `GET /feed.xml` when a corresponding `.chaptermarks.txt` sidecar file exists in `OUTPUT_DIR`.
*   Maintain the existing cache-invalidation rules for `feed.xml` so that the feed rebuilds when chapter sidecars are modified or added.

**Non-Goals:**
*   Add any frontend user interface (UI) to the podcast landing page.
*   Directly parse M4A binary files for chapters (which would require external tools like `ffprobe` or parsing libraries). Only sidecar `.chaptermarks.txt` files are parsed.
*   Modify `local/chapters.py` or the local post-processing pipeline.

## Decisions

### 1. Endpoint Path and Resolution Strategy
We will expose `GET /chapters/{filename}` where `{filename}` is expected to end in `.json` (e.g. `GET /chapters/<hash>.json`). 
*   **Resolution:** The server resolves the request by looking for a sibling file with the same base name but with a `.chaptermarks.txt` extension in `OUTPUT_DIR`. E.g., request `/chapters/9a8b7c.json` will read `/OUTPUT_DIR/9a8b7c.chaptermarks.txt`.
*   **Security:** Suffix and traversal validation will be performed using the existing `_safe_resolve` helper.

### 2. Chapters JSON Structure
The API will return the Podcasting 2.0 Chapters standard structure.
*   **Format:**
    ```json
    {
      "version": "1.2",
      "chapters": [
        {
          "startTime": 0.0,
          "title": "Introduction"
        },
        {
          "startTime": 120.5,
          "title": "Chapter 2"
        }
      ]
    }
    ```
*   **Unit of Time:** `startTime` is represented as floating-point seconds (calculated as `start_ms / 1000.0`), which is the most widely supported format across modern podcast clients.

### 3. FFMETADATA1 Parser Implementation
We will implement a line-by-line parser for FFMETADATA1 metadata in `cloud/app.py`.
*   The parser will look for `[CHAPTER]` markers.
*   Each chapter will parse `START=<ms>` and `title=<escaped_title>`.
*   It will unescape standard FFmpeg escape sequences (e.g., `\\`, `\=`, `\;`, `\#`, `\n`) back to plain characters.
*   It will validate that timestamps are non-negative and sequential.

### 4. Cache Invalidation Integration
We will update the cached feed validation check to monitor not only `.m4a` files but also `.chaptermarks.txt` files. 
*   When a new chapter marks file is added or an existing one is updated, the feed cache will be invalidated so that `<podcast:chapters>` elements are dynamically updated in the RSS output.

## Risks / Trade-offs

*   **Risk:** Invalid, empty, or corrupted `.chaptermarks.txt` files in `OUTPUT_DIR`.
    *   **Mitigation:** The parser will handle exceptions gracefully. If a file is unreadable or malformed, the server logs a warning and returns a `404 Not Found` response rather than crashing or serving broken JSON.
*   **Risk:** Suffix mismatch in clients requesting `.json` when the physical file is `.chaptermarks.txt`.
    *   **Mitigation:** The route pattern specifically matches `{filename}` ending with `.json` and maps it directly to the `.chaptermarks.txt` file, ensuring clean path routing.
